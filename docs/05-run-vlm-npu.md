# 05 — Run VLM on NPU: MobileCLIP + Tiny Bridge

How to run the MobileCLIP-S0 vision encoder and tiny VLM bridge on the
A733 NPU, producing an image embedding and proving the vision-to-text
data path.

## What this achieves

- MobileCLIP-S0 vision encoder: `1×3×256×256` image → `1×512` embedding
  at 22.6 ms on the NPU
- Tiny VLM bridge: image embedding + token window → logits (proves the
  NPU data path for image projector + decoder)
- Validated: vs-host cosine 0.99996 (encoder) and 0.99999 (bridge)
  with top-5 match

**Note**: the bridge decoder has a vocabulary of only 16 tokens. This is not
a usable captioning or VQA model — it proves that the NPU can execute the
full VLM graph path: vision encoder → projector → token Gather → concat →
decoder → logits.

## Prerequisites

- Host: [01-setup-host.md](01-setup-host.md) completed
- Board: [02-board-bringup.md](02-board-bringup.md) completed
- `vpm_run` available on the board
- A test image (the existing `dog_224_224.jpg` from the SDK works)

## Step 1: Prepare the test image tensor

On the host:

```bash
python3 scripts/host/make_mobileclip_input.py \
  --image work/ai-sdk/ZIFENG278-ai-sdk/examples/resnet50/input_data/dog_224_224.jpg \
  --output work/generated/b3_mobileclip_s0_vision/pixel_values.npy
```

This produces a `1×3×256×256` tensor with MobileCLIP preprocessing
(rescale 1/255, no normalization).

## Step 2: Convert MobileCLIP-S0 vision encoder to NBG

The MobileCLIP-S0 ONNX is from `Xenova/mobileclip_s0` on Hugging Face.
The ONNX was previously fixed to static `1×3×256×256` input shape.

```bash
DOCKER_RUN_ARGS="--cpus 10 --memory 24g" \
  scripts/host/convert_onnx_to_nbg.sh \
    --name b3_mobileclip_s0_vision_int16 \
    --onnx path/to/mobileclip_s0_vision.onnx \
    --dataset work/generated/b3_mobileclip_s0_vision/pixel_values.npy \
    --quant int16 \
    --inputs pixel_values \
    --input-size-list 1,3,256,256 \
    --outputs attach_image_embeds/out0
```

Expected: `Error(0),Warning(0)`, NBG size ~19.4 MB.
Output: `1×512` int16 dynamic fixed point, `fl=17`.

## Step 3: Convert tiny VLM bridge to NBG

The VLM bridge ONNX is generated by `scripts/host/make_tiny_vlm_bridge_onnx.py`.
It accepts a `1×512` image embedding (`dfp=16`) and `1×4` int32 tokens, and
produces `1×5×16` logits through the full decoder path.

```bash
# Generate the bridge ONNX
python3 scripts/host/make_tiny_vlm_bridge_onnx.py \
  --output-dir work/generated/b3_tiny_vlm_bridge

# Convert to int16 NBG
scripts/host/convert_onnx_to_nbg.sh \
  --name b3_tiny_vlm_bridge_int16 \
  --onnx work/generated/b3_tiny_vlm_bridge/tiny_vlm_bridge.onnx \
  --dataset work/generated/b3_tiny_vlm_bridge/dataset.txt \
  --quant int16 \
  --inputs image_embed,token_ids \
  --input-size-list 1,512 "1,4" \
  --outputs attach_logits/out0
```

Expected: `Error(0),Warning(0)`, NBG size ~94 KB.
Output: `1×5×16` int16 dynamic fixed point, `fl=14`.

### Multi-input dataset note

The VLM bridge has two inputs (`image_embed` + `token_ids`). The dataset
file must reference two separate data files. The conversion wrapper handles
this automatically when you pass comma-separated input names and the dataset
points to the two `.npy`/`.txt` files.

## Step 4: Upload to board

```bash
# Upload MobileCLIP NBG
scp -r work/model-packages/b3_mobileclip_s0_vision_int16/int16/* \
  user@board:~/a733_npu_driver/models/b3_mobileclip_s0_vision_int16/

# Upload bridge NBG
scp -r work/model-packages/b3_tiny_vlm_bridge_int16/int16/* \
  user@board:~/a733_npu_driver/models/b3_tiny_vlm_bridge_int16/

# Upload runner script
scp scripts/board/run-b3-vpm-package.sh \
  user@board:~/a733_npu_driver/scripts/board/
```

## Step 5: Run vision encoder on board

```bash
cd ~/a733_npu_driver

bash scripts/board/run-b3-vpm-package.sh \
  --model-dir models/b3_mobileclip_s0_vision_int16 \
  --log-dir logs/board/b3-mobileclip \
  --vpm-run /opt/vpm_run/vpm_run \
  --vip-lib /home/orangepi/lib \
  --loops 5
```

Expected output:
```
VIPLite driver software version 2.0.3.2-AW-2024-08-30
cid=0x1000003b
vpm run ret=0
profile mean 22,605.8us
memory pool 1,573,888 bytes
```

The embedding is saved to `logs/board/b3-mobileclip/output_0.txt`.

### Verify against host reference

```bash
python3 scripts/host/compare_outputs.py \
  --golden logs/host/b3-mobileclip-oracle/output_0.txt \
  --board logs/board/b3-mobileclip/output_0.txt \
  --output-json logs/host/b3-mobileclip-vs-board.json
```

Expected: cosine ~0.99996, top-5 match yes.

## Step 6: Run VLM bridge end-to-end

The bridge needs the board-produced MobileCLIP embedding as its image input.

```bash
# Pack the board embedding into the bridge input format
python3 scripts/host/pack_nbg_input_from_text.py \
  --meta models/b3_tiny_vlm_bridge_int16/nbg_meta.json \
  --input-index 0 \
  --text logs/board/b3-mobileclip/output_0.txt \
  --output models/b3_tiny_vlm_bridge_int16/input_0.dat

# Run the bridge NBG
bash scripts/board/run-b3-vpm-package.sh \
  --model-dir models/b3_tiny_vlm_bridge_int16 \
  --log-dir logs/board/b3-tiny-vlm-bridge \
  --vpm-run /opt/vpm_run/vpm_run \
  --vip-lib /home/orangepi/lib \
  --loops 20
```

Expected output:
```
cid=0x1000003b
vpm run ret=0
profile mean 62.6us
wall mean 140.4us
peak RSS 1,792 KB
```

## Step 7: Verify bridge accuracy

```bash
python3 scripts/host/compare_outputs.py \
  --golden logs/host/b3-bridge-oracle/output_0.txt \
  --board logs/board/b3-tiny-vlm-bridge/output_0.txt \
  --output-json logs/host/b3-bridge-vs-board.json
```

Expected: full-tensor cosine ~0.99998, last-token cosine ~0.999999,
last-token top-1 match, top-5 index match yes.

## Results summary

| Component | Input | Output | Latency | Peak RSS | NBG size | Cosine vs ACUITY |
|---|---|---|---|---|---|---|
| MobileCLIP-S0 | 1×3×256×256 | 1×512 | 22.6 ms | 14 MB | 19 MB | 0.99996 |
| Tiny VLM bridge | embed + 4 tokens | 1×5×16 logits | 0.063 ms | 2 MB | 94 KB | 0.99999 |

## Limitations

- Image decode/resize/rescale is **CPU-side** preprocessing (not on NPU)
- The bridge decoder is tiny (vocab=16) — it proves the data path, not a
  usable model
- A real VL-NPU model (nanoLLaVA, SmolVLM class) would need:
  - A real projector (not the tiny one)
  - A much larger decoder NBG
  - Same fixed-window constraints as the LLM path
  - Potential int16 quality risks seen on larger models
