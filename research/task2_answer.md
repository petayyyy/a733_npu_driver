# Running Qwen2.5-0.5B-Instruct on the Allwinner A733 / Vivante VIP9000 NPU: A First-Principles Deployment Assessment

## TL;DR
- **Coherent Qwen2.5-0.5B with all transformer compute running on the VIP9000 NPU is NOT demonstrated anywhere publicly today, and is not achievable as a turnkey path with public tools.** VeriSilicon's own ACUITY model zoo *lists* Qwen2.5-0.5B-Instruct as a convertible model, but there is no published tokens/sec, no KV-cache documentation, and every real end-to-end LLM demo on VIP9000 hardware (NXP i.MX 8M Plus, Amlogic A311D) runs the LLM on the **CPU**, not the NPU. *(verified)*
- **The single most-likely-to-work path is VeriSilicon ACUITY → NBG → VIPLite using a prefill/decode split with a fixed context window and W8A16 quantization (per-channel INT8 weights + INT16/FP16 activations), offloading the heavy matmuls (QKV, MLP, output projection) to the NPU while the CPU orchestrates the autoregressive loop, KV-cache, RoPE, sampling, embedding/LM-head, and likely attention softmax.** This is a build-it-yourself integration, not a supported product; expect weeks-to-months of engineering and likely direct VeriSilicon/Allwinner support. *(assumption, grounded in verified toolchain capabilities)*
- **Decode is memory-bandwidth-bound, not TOPS-bound.** At the A733's 32-bit LPDDR4/4x/5 (up to LPDDR5, ~16–25 GB/s), a 0.5B INT8 model is capped at roughly 8–25 tok/s and INT4 at ~15–40 tok/s whether or not the NPU does the matmuls — so the NPU primarily buys energy efficiency and CPU-offload, not dramatic speed. A "few tokens/sec, coherent" goal is physically realistic; getting the compute *onto the NPU coherently* is the hard part. *(assumption from verified bandwidth figures)*

## Key Findings

1. **No public proof of LLM decode on VIP9000.** An exhaustive search of GitHub, NXP/Khadas forums, VeriSilicon docs, blogs and papers found NO public demonstration of any LLM generating coherent text with its compute on a VIP9000 NPU. The disconfirming evidence is direct: NXP's own eIQ GenAI Flow Demonstrator README (GitHub `nxp-appcodehub/dm-eiq-genai-flow-demonstrator`) states verbatim: *"For i.MX8MP EVK, the NPU Acceleration is unavailable. LLM pipeline will run on CPUs for this platform."* NXP's NPU-accelerated LLM work is an experimental feature targeting only the *Neutron* NPU in the i.MX 95 (per NXP UG10166 i.MX Machine Learning User's Guide, Rev. LF6.18.2_1.0.0, 26 March 2026) — a completely different, NXP in-house NPU, not VeriSilicon. *(verified)*

2. **VeriSilicon does list Qwen2.5-0.5B as a target.** The VeriSilicon Acuity Model Zoo (`verisilicon.github.io/acuity-models`, "LLM-Chat" section) explicitly includes *"Qwen2.5-0.5B-Instruct · Qwen2.5-1.5B-Instruct · Qwen2.5-3B-Instruct · Qwen3-0.6B · MiniCPM-1B-sft-bf16 · TinyLlama-1.1B · Llama3.2-1B-Instruct · gemma-3-270m-it"* and others. This is a model-conversion support list, not a benchmarked demonstration. It is meaningful — it signals VeriSilicon has at least internally exercised these models through ACUITY — but no recipe, throughput, or graph structure is published. *(verified)*

3. **The toolchain is a static, fixed-shape graph compiler.** ACUITY / acuitylite imports ONNX/TF/PyTorch/TFLite and emits a "Network Binary Graph" (NBG) plus TIM-VX cases. Per the Radxa Cubie (A733) Vivante SDK docs, it supports UINT8, per-channel INT8 (PCQ), INT16, BF16 and mixed quantization, with automatic operator fusion. The VIP9000 hardware natively supports INT8/INT16/FP16/BF16. NBG runs on either the heavier Unified/OpenVX driver or the lightweight VIPLite driver. Critically, the VIPLite driver supports **Multi-Graph** execution and an MMU — so chaining multiple NBGs and passing tensors between them at runtime IS supported by the runtime. *(verified)*

4. **No public KV-cache / dynamic-shape support for LLMs.** There is no public VeriSilicon documentation describing a prefill-NBG/decode-NBG split, stateful tensors, or a variable-shape KV-cache. The public toolchain produces static fixed-shape graphs. This is the central technical obstacle: autoregressive decode on a static-graph NPU must use a fixed context window with pre-allocated, fixed-size KV tensors (write-at-position), or recompute — exactly the pattern documented for other static-shape NPUs (Intel's OpenVINO NPU pipeline caps prompts at a fixed length such as 1024 tokens; the mllm-NPU paper chunks prompts into fixed sizes because NPUs don't natively support dynamic shapes). *(toolchain limits verified; the prefill/decode design is an assumption by analogy)*

5. **ONNX Runtime VSI EP and TVM vsi_npu BYOC are both immature for this.** The NXP ONNX Runtime VSI NPU execution provider has effectively no quantized-op support — an NXP engineer states verbatim on the NXP Community forum: *"According reply from R&D, VSI NPU execution provider in ONNX support for quantized models is very limited (I would say none)."* The VeriSilicon TVM `vsi_npu` BYOC backend exists and compiles NBGs but has open issues even for MobileNet on real VIM3/i.MX8 hardware (the recurring `"PLS isn't existed"` runtime failure). Neither is a credible LLM path today. *(verified)*

6. **The etnaviv/Teflon open-source driver is vision-only.** Tomeu Vizoso's reverse-engineered Etnaviv NPU driver (mainline Mesa/Linux 6.9+) accelerates convolutions/MobileNet/object detection on the A311D and i.MX 8M Plus VIP9000 (object detection at 30 fps on the 8M Plus). It does NOT implement the transformer/LLM operator set (attention, softmax, RMSNorm, RoPE), and when explicitly asked "Is your driver able to run even LLM models?", the response pointed only to TFLite vision examples. Not viable for LLMs. A community Frigate port for the A733 (GitHub `blakeblackshear/frigate` Discussion #23418) likewise confirms vision-only NPU use via *"VIPLite 2.0... chip/optimize ID 0x1000003B, /dev/vipcore"* with a note that *"int16 quantization is worth it over uint8"* — but no LLM decode. *(verified)*

7. **Rockchip RKLLM is the proof-of-concept reference — but a different NPU.** Rockchip's RKLLM stack (RK3576/RK3588, NOT VeriSilicon) demonstrably runs Qwen2/Qwen2.5, Llama, Phi-2, and DeepSeek-R1-Distill-Qwen-1.5B with W4A16 and W8A8 quantization, a real KV-cache (including incremental/prompt-cache decode), and usable throughput. It proves the *concept* of a vendor small-LLM-on-NPU stack with a purpose-built runtime and a true KV-cache. VeriSilicon has no public equivalent; the gap is software, not silicon. *(verified)*

## Details

### The hardware reality
- The Allwinner A733 integrates a VIP9000-series NPU (optional, ~3 TOPS INT8), alongside an octa-core CPU (2×Cortex-A76 @ 2.0 GHz + 6×Cortex-A55 @ 1.8 GHz), an Imagination BXM-4-64 GPU, and a RISC-V XuanTie E902 @ 200 MHz, with a 32-bit LPDDR4/LPDDR4x/LPDDR5 interface up to 16 GB (per Radxa A733 Datasheet V0.93 / CNX Software, Dec 6 2024). The NPU exposes chip/optimize ID 0x1000003B on /dev/vipcore via VIPLite. *(verified)*
- This is the same NPU family as the NXP i.MX 8M Plus (2.3 TOPS, 32-bit LPDDR4 ≈ 16 GB/s theoretical) and Amlogic A311D (5 TOPS). Precedents from those platforms transfer in principle, but the A733 uses a newer VIP9000 revision. *(verified for the family relationship; transfer is an assumption)*
- **Decode is memory-bound.** During autoregressive decode you stream the entire weight set once per token. At ~16 GB/s (i.MX 8M Plus LPDDR4) to ~25 GB/s (A733 LPDDR5), at realistic ~50% bandwidth efficiency, a 0.5B INT8 (~500 MB) model is bounded to roughly 8–25 tok/s and INT4 (~250–350 MB) to ~15–40 tok/s. The 3 TOPS compute is NOT the bottleneck. Real-world sanity check: per arXiv:2603.23640, a Raspberry Pi 5 + Hailo-10H NPU (40 TOPS) *"sustains 6.9 tok/s at under 2 W with near-zero variance"* and is *"limited by on-module memory bandwidth"* — and independent testing (CNX Software / schwab.sh) found the Pi 5 CPU sometimes matches or beats the Hailo-10H on decode, confirming that TOPS is not the limiter — bandwidth is. *(bandwidth/Hailo figures verified; A733 tok/s ceilings are calculated assumptions)*

### Why Qwen2.5 specifically is hard
- **Activation outliers.** Qwen2.5, like most modern LLMs, has large activation outliers that wreck naive per-tensor INT8 *activation* quantization (SmoothQuant's own data shows multi-point accuracy drops on Qwen-class models with per-tensor INT8). W8A8 (INT8 weights AND activations) is therefore risky for coherence; **W8A16** (per-channel INT8 weights, INT16 or FP16 activations) is the safer NPU-friendly choice, and the VIP9000 natively supports INT16 and FP16 activations. *(verified hardware support; outlier sensitivity is well-established)*
- **Operator coverage.** RMSNorm, RoPE, SwiGLU/SiLU, and the attention softmax must map to NBG-supported ops or run on CPU. VeriSilicon claims 120+ operators and "Transformer" support (BERT, ViT, Swin are in the zoo), but the autoregressive-specific machinery (causal masking, KV concatenation, position-dependent RoPE) is not a documented turnkey op set. *(verified that transformer encoders are supported; decode machinery coverage is unverified)*
- **Vocab/embedding size.** Qwen2.5 uses a ~151,936-token vocabulary (per HF transformers Qwen2 config default `vocab_size=151936`; the Qwen2.5 0.5B has 24 layers, hidden size 896, 14 query / 2 KV heads). This large vocab makes the embedding and final LM-head projection heavy; these are typically kept on CPU or run as separate gather/matmul stages. By contrast, Llama-3.2/Gemma vocabularies are smaller, easing this cost. *(verified)*

### The right graph granularity
Given a static-shape compiler with multi-graph runtime support, the architecturally correct decomposition is:
- **Embedding/LM-head on CPU** (large, gather-heavy, vocab-bound).
- **A prefill NBG** compiled for a fixed prompt length (e.g., 128/256/512 tokens) producing the initial KV-cache.
- **A decode NBG** compiled for sequence length 1 with a fixed-maximum context window, reading/writing pre-allocated fixed-size KV tensors at a runtime position index.
- **Whole-stack vs per-block:** one decode NBG covering all 24 blocks minimizes per-token dispatch overhead and is preferable if it compiles. Per-block chaining (relying on VIPLite Multi-Graph) is the fallback if the monolithic graph exceeds compiler/network-size limits (the i.MX 8M Plus has a documented network-size limit of 2048 referenced in its reference manual).
- **KV-cache** must be a fixed, pre-allocated tensor (no dynamic growth). Realistic context length on this device class is short — on the order of 512–2048 tokens — bounded by both the fixed-shape requirement and memory.
- **Attention softmax / RoPE / RMSNorm:** attempt on-NPU; fall back to CPU per-op if ACUITY can't compile them with acceptable accuracy. *(graph design is an assumption grounded in verified static-graph + multi-graph capabilities)*

### Ranked viable strategies

**Strategy A — Most likely to work: ACUITY → NBG → VIPLite, prefill/decode split, W8A16, CPU orchestration.**
- *Toolchain:* ACUITY Toolkit (or open `acuitylite` + the matching Vivante SDK/licence for the A733 chip ID 0x1000003B) to compile two NBGs; VIPLite runtime on /dev/vipcore to execute them; CPU (llama.cpp-style C/C++) for tokenization, embedding, RoPE, KV management, sampling.
- *Quantization:* per-channel INT8 weights, INT16/FP16 activations (W8A16); optionally apply AWQ/SmoothQuant offline so weights quantize cleanly before ACUITY import.
- *Graph:* fixed-length prefill NBG + seq-len-1 decode NBG with fixed-size KV tensors; embedding/LM-head on CPU.
- *Effort:* high (weeks–months); requires the licensed VeriSilicon SDK for the exact chip, and probably direct VeriSilicon/Allwinner engineering support for the LLM decode pattern.
- *Precedent:* VeriSilicon lists Qwen2.5-0.5B in the zoo (support, not demo). Strongest available, but unproven on real VIP9000 silicon publicly.

**Strategy B — Pragmatic fallback that IS proven to run (just not on the NPU): CPU inference via llama.cpp, NPU idle.**
- This is exactly what NXP ships for the i.MX 8M Plus eIQ GenAI Flow today: the LLM runs on the Cortex-A cores. On the A733's A76/A55 cores, a 0.5B INT4/INT8 GGUF will produce coherent Qwen2.5-0.5B output at several tok/s. It fails the "compute on the NPU" requirement but is the only path *guaranteed* coherent and runnable today.
- *Effort:* low. Use as the bring-up baseline and correctness oracle.

**Strategy C — Partial NPU offload: TIM-VX / tflite-vx-delegate for the matmul-heavy blocks only.**
- Build the transformer as TFLite and run via the VX delegate so supported ops (fully-connected/matmul, conv) hit the NPU while unsupported ops fall back to CPU. The most "incremental" way to get *some* transformer compute onto the NPU.
- *Risk:* heavy CPU fallback for attention/softmax/RoPE; the delegate was built for vision graphs and quantized-LLM op coverage is unproven. Likely lands as a hybrid where only linear layers are accelerated.
- *Effort:* medium; good as an experiment to measure how much actually offloads.

**Strategy D — Not recommended today: TVM vsi_npu BYOC, ONNX Runtime VSI EP, or etnaviv/Teflon.**
- All three are vision-only (etnaviv), quantization-immature (ONNX RT VSI EP, "very limited (I would say none)"), or buggy even on MobileNet (TVM vsi_npu "PLS isn't existed"). None is LLM-ready on VIP9000. Track for the future; don't bet a deployment on them.

### The single best recommended approach + first steps
**Pursue Strategy A, but stand up Strategy B first as the correctness baseline, and de-risk in this order:**
1. **Baseline:** Run Qwen2.5-0.5B-Instruct INT4/INT8 GGUF on the A733 CPU via llama.cpp. Confirm coherence; record CPU tok/s and exact token outputs as your reference oracle.
2. **Toolchain bring-up:** Obtain the licensed ACUITY/VIPLite SDK for chip ID 0x1000003B from Allwinner/VeriSilicon. Compile and run a trivial NBG (single matmul, then MobileNet) on /dev/vipcore via VIPLite to prove the offline→NBG→hardware path end-to-end.
3. **Single-block proof (make-or-break):** Export ONE Qwen2.5 transformer block to ONNX with fixed shapes (seq-len-1, fixed KV window), quantize W8A16, compile to NBG, and validate its output numerically against the CPU reference for that block. This tells you whether RMSNorm/RoPE/softmax/attention compile with acceptable accuracy or must move to CPU.
4. **Decode loop:** Wrap the per-block (or whole-stack) decode NBG in a CPU autoregressive loop with a fixed-size KV-cache, embedding and LM-head on CPU, greedy sampling. Compare generated tokens against the oracle.
5. **Prefill + scale:** Add a fixed-length prefill NBG, then push context length and measure tok/s. Engage VeriSilicon support the moment step 3 reveals op-coverage or accuracy gaps.
- **Benchmarks that change the plan:** If step 3 shows attention/RMSNorm/RoPE won't compile coherently in ACUITY, fall back to Strategy C (linear-layers-only offload) or accept Strategy B. If measured NPU decode tok/s is no better than CPU (likely, since decode is bandwidth-bound), the NPU's only remaining value is freeing the CPU and saving power — decide whether that justifies the integration cost.

### If Qwen2.5-0.5B proves impractical: the best proven alternative
There is no *proven* coherent small-LLM-on-VIP9000 today, so "proven" must be relativized:
- **On a VIP9000, the lowest-risk model bets** (still unproven publicly) are the ones VeriSilicon explicitly lists AND that are quantization-friendly: **gemma-3-270m-it, MiniCPM-1B, Llama-3.2-1B, or TinyLlama-1.1B.** A simpler architecture with fewer activation outliers and a smaller vocab (Llama/Gemma vocabs are far smaller than Qwen's ~152k) quantizes more cleanly to NPU-friendly formats and cuts embedding/LM-head CPU cost. **gemma-3-270m-it** is the smallest and most likely to fit and run at the best tok/s.
- **If the requirement is "coherent small LLM physically running its matmuls on an NPU, proven on real silicon," the only proven stack in this device class is Rockchip RKLLM on an RK3576/RK3588** — a different SoC and NPU entirely. If hardware choice is even slightly flexible, an RK3576/RK3588 board runs Qwen2.5-class models on its NPU today with W4A16/W8A8 and a real KV-cache. On the A733/VIP9000 specifically, expect to do the integration yourself.

## Recommendations
1. **Immediately:** Stand up the llama.cpp CPU baseline (Strategy B) on the A733 to lock in a coherent reference and realistic tok/s. This is your oracle and your fallback.
2. **In parallel:** Acquire the licensed ACUITY + VIPLite SDK for chip ID 0x1000003B and prove the NBG→/dev/vipcore path with MobileNet, then a single matmul.
3. **Gate the whole project on the single-transformer-block NBG accuracy test (Strategy A, step 3).** This cheaply reveals whether the NPU can host attention/RMSNorm/RoPE coherently. Do not invest in the full decode loop until this passes.
4. **Quantize W8A16, not W8A8.** Use per-channel INT8 weights with INT16/FP16 activations; pre-condition weights with AWQ or SmoothQuant offline. Reserve W8A8/INT4 experiments for after coherence is established.
5. **Engage VeriSilicon/Allwinner support early.** Given zero public LLM-decode precedent and no public KV-cache documentation, vendor help is likely required to reach coherent NPU-resident decode. Ask specifically for: (a) any internal Qwen/Llama NBG example, (b) the recommended prefill/decode graph structure, (c) whether VIPLite supports a stateful KV tensor pattern.
6. **Decision threshold:** If after the block test the NPU path needs vendor code you can't get, or NPU decode tok/s ≤ CPU tok/s with no power/CPU-offload benefit that matters for your product, ship Strategy B (CPU) — or switch silicon to an RK3576/RK3588 if NPU-resident LLM compute is a hard requirement.

## Verdict
**Coherent Qwen2.5-0.5B with compute on the VIP9000 NPU is not achievable as a proven, public, turnkey recipe today — it requires either substantial self-integration (Strategy A) or vendor support, and there is zero public evidence anyone has done it.** The chip vendor (NXP) itself runs LLMs on the CPU for this exact NPU family. The closest *proven* coherent small-LLM-on-NPU deployments in this device class are on a different NPU (Rockchip RK3576/RK3588 via RKLLM). For the A733 specifically: the realistic, low-risk way to get coherent Qwen2.5-0.5B *running now* is CPU (llama.cpp); the realistic way to get *compute onto the NPU* is the ACUITY→NBG→VIPLite prefill/decode build, gated on a single-transformer-block accuracy proof, with vendor support on standby.

## Caveats
- **"Listed in the ACUITY zoo" ≠ "demonstrated coherent on hardware."** VeriSilicon publishes no tokens/sec, recipe, or KV-cache docs for any LLM. Treat the Qwen2.5-0.5B listing as evidence of intent/internal testing, not a working public recipe.
- **The decode-on-static-graph KV-cache problem is unsolved publicly for this toolchain.** The prefill/decode + fixed-window design here is an inference from how comparable static-shape NPUs (OpenVINO NPU, mllm-NPU) handle it, NOT a documented VeriSilicon capability — tagged assumption.
- **Bandwidth and tok/s ceilings are calculated** from verified interface widths/clocks at ~50% efficiency; real throughput depends on the specific A733 board's DRAM (LPDDR4 vs LPDDR5) and contention — tagged assumption.
- **i.MX 8M Plus / A311D precedents transfer "in principle"** — the A733 uses a newer VIP9000 revision; op coverage and compiler behavior may differ. NXP's "NPU acceleration unavailable for LLM" is specific to NXP's BSP/integration, not necessarily a hardware impossibility — but it is the most direct available signal that this is hard even for the chip vendor.
- **Source dating:** key NXP/VeriSilicon docs cited are current as of 2026; the etnaviv LLM-incapability is current as of mid-2025 and could change as Teflon matures.
