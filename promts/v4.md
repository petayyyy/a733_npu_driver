PART A — the application (prefer a simple web UI; CLI acceptable as a fallback):
1. A local app (Python, e.g. FastAPI/Gradio or a small Flask + static page) that runs on the Orange
   Pi and serves a browser UI on the LAN. Features:
   - Drag/drop or pick an image, type a question, get the model's answer.
   - Stream the answer token-by-token; show generation tok/s and which backend is active.
   - A backend selector: "CPU (SmolVLM-256M, fast, recommended)" and, IF available, "NPU vision +
     CPU LLM (frees CPU cores, slower vision)". Default to CPU.
   - A model selector: SmolVLM-256M (default) and SmolVLM-500M (more detail, slower); do NOT offer
     >1B models (InternVL3.5-1B loads but is too slow and starves RAM — V3 verified; exclude it).
   - Show RAM/CPU usage and a one-line note that ROS2 headroom stays free with the 256M model.

   CPU path; the V2c/V2d NPU encoder + injector for the offload path (only expose that option if the
   V2d e2e gate passed — otherwise hide it and note "experimental").
3. Keep it lightweight (the board runs ROS2 too): minimal deps, configurable port, a --cpu-only
   flag, and sane defaults.

PART B — install & deployment guide (docs/09-vlm-app.md):
1. Step-by-step from a fresh Orange Pi Zero 3W (image 1.0.4 / the working kernel): system deps,
   building/installing llama.cpp with multimodal (mmproj) support, downloading the SmolVLM-256M
   GGUF + mmproj (exact HF links/filenames already used in the repo), and the NPU bring-up only if
   the user wants the offload mode (link docs/02-board-bringup.md).
2. How to launch the app, open the UI (board-IP:port), and a troubleshooting section (OOM if a too-
   big model is picked, port conflicts, NPU not present → CPU fallback).
3. A short "hardware requirements / expectations" box: SmolVLM-256M ~634 MB RSS, 52 tok/s, leaves
   >5 GB for ROS2/picoclaw; 500M as the higher-detail option; >1B not supported on 6 GB.
4. A "for other A733 boards" note: the CPU path is board-agnostic (any A733/aarch64 with enough
   RAM); the NPU-offload path needs /dev/vipcore + the glibc-matched VIPLite .so (link the porting
   guide).
