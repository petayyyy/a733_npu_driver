# T7 W8A16 Qwen Host Gate

Date: 2026-06-23

## Purpose

Verified: task T7 tested the W8A16 hypothesis for Qwen2.5-0.5B-Instruct:
keep token embedding, attention activations, softmax, RMSNorm, RoPE, lm_head,
and logits at int16, while quantizing only transformer linear/MLP
`fullconnect` weights to per-channel int8.

Verified: all T7 validation in this report is host-only. The A733 board was
not used because the GO/NO-GO host cosine gate did not pass.

## Code Changes

Verified code changes:

- `scripts/host/make_real_llm_onnx.py`
  - added `--smoothquant-scales` for exact SmoothQuant-style
    re-parameterization before ONNX export;
  - added `--debug-layer-outputs` for per-layer host cosine checks.
- `scripts/host/make_real_llm_smoothquant_scales.py`
  - collects activation maxima with torch inside `ubuntu-npu:v2.0.10.1`;
  - writes `layerN.attn_norm`, `layerN.ctx`, `layerN.mlp_norm`, and
    `layerN.gated` scale vectors.
- `scripts/host/make_w8a16_quantize.py`
  - builds a seeded ACUITY `.quantize` table from an int16 import;
  - keeps activation/output qparams as int16;
  - replaces only transformer `fullconnect:{weight,bias}` entries with
    per-channel int8/int32 entries;
  - leaves the tied logits/lm_head projection int16 by default;
  - supports `--force-int16-layers` for host-side widening.
- `scripts/host/dump_real_llm_oracle.py`
  - dumps FP32 oracle logits and optional per-layer outputs for a fixed token
    window.
- `scripts/host/compare_acuity_host_to_oracle.py`
  - compares packaged ACUITY host outputs against the FP oracle and writes JSON
    metrics.

```text

```

Conclusion: quantizing Qwen layer0 transformer fullconnects is already below
the T7 per-layer threshold. The layer-widening mechanism works mechanically.

## Result

Verified: W8A16 as tested here is a host-side no-go. Because the host gate did
not pass, the A733 board was not touched.

Verified production status is unchanged: the coherent path remains the existing
int16 NPU path for models that fit board RAM. Full Qwen2.5-0.5B still does not
fit the current 1GiB board runtime even when compressed by previous seeded PCQ
work, and this T7 W8A16 path did not produce a host-quality package worth
uploading.

## Next

Assumption: a less aggressive SmoothQuant configuration might avoid breaking
the int16 control. The next host-only experiment, if T7 is continued, should
try narrower smoothing such as lower `alpha`, tighter `scale_max`, or smoothing
only selected projections. The board should still remain out of the loop until
the host logits/per-layer cosine gate passes.

Before more ACUITY runs, free disk space on `C:` should be recovered or the
large ignored `work/` artifacts should be moved to a drive with more space.
